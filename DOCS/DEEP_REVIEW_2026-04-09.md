# 인싸이트 오더 V3 심층 보고서

작성일: 2026-04-09

## 1. 요약

이 코드베이스는 "행사 기간에만 운영되는 저볼륨 B2B 주문 시스템"이라는 문제를 꽤 정확하게 풀고 있다. 고객 주문 플로우는 짧고, 관리자 화면은 운영 실무에 맞춰 빠르게 확장되어 왔으며, Supabase를 중심으로 프론트엔드와 백엔드가 비교적 단순한 형태로 연결되어 있다. 특히 `site_settings`, 토큰 기반 주문 조회, 현장판매 토글, 알림톡 연동, 주문 묶음 처리 등은 실제 운영 맥락을 반영한 결과물이다.

반면 현재 가장 큰 문제는 "구현된 권한 모델", "DB RLS 정책", "프론트 권한 추론"이 서로 다른 언어를 쓰고 있다는 점이다. 이 불일치는 단순한 코드 냄새가 아니라, 운영자 접근 실패 또는 잘못된 권한 부여로 이어질 수 있는 구조적 결함이다. 여기에 주문/배송 정책이 여러 계층에 분산되어 있고, 일부 Edge Function이 실패를 `200 OK`로 감추며, 테스트 스위트도 현재 상태를 충분히 방어하지 못한다. 제품은 이미 유용하지만, 다음 단계로 가기 전에 권한/정책/검증 계층을 먼저 정리해야 한다.

## 2. 구조 평가

프론트엔드는 React + MUI + Supabase JS 조합으로 일관되어 있고, 고객용 주문 플로우와 관리자용 운영 화면이 명확히 분리되어 있다. 고객 주문은 `OrderPage`에서 시작해 Edge Function `create-order`로 들어가고, 주문 상태 조회는 `access_token` 기반 RPC로 닫아둔 점이 좋다. 관리자 측은 주문, 이벤트, 상품, 대시보드, 사용자 관리가 분리되어 있고, 운영 편의 기능도 충분히 많다.

DB와 서버리스 계층은 Supabase에 집중되어 있어 배포와 운영 복잡도는 낮다. `site_settings`를 통해 배송 정책 일부를 DB로 끌어올린 방향은 옳다. 다만 실제 비즈니스 규칙은 아직 프론트, API 모듈, Edge Function에 동시에 남아 있어서 "설정 가능해 보이지만 사실은 군데군데 하드코딩된 시스템"에 가깝다.

## 3. 핵심 강점

- 주문 조회를 숫자 ID가 아닌 `access_token` 기반으로 막은 점은 공개 주문 조회 시스템으로서 올바른 선택이다.
- 배송비/무료배송 기준을 `site_settings`로 이동시키기 시작한 것은 운영 시스템화의 좋은 출발이다.
- 관리자 기능이 실제 현장 흐름에 가깝다. 신규 주문 추가, 상태 일괄 변경, 엑셀 다운로드, 필드 리포트, 실시간 주문 알림 등은 실무 친화적이다.
- `create-order`가 서버에서 가격을 재계산하는 구조는 신뢰 경계를 잘 잡고 있다.

## 4. 주요 리스크

### 4-1. 권한 모델 불일치

프론트는 `orders:view`, `products:view`, `events:view` 같은 네임스페이스형 권한을 사용한다. 그런데 DB RLS는 `view`, `edit`, `master`를 기대한다. 이 상태라면 일반 운영자는 프론트에서 메뉴가 보여도 DB 정책에서 막히거나, 반대로 프론트가 임의 기본 권한을 부여해 화면을 열어버릴 수 있다. 근거:

- `AuthContext`의 기본 권한 추론: `inpsyt-order-frontend/src/AuthContext.jsx`
- RLS의 `view`/`edit` 정책: `supabase/migrations/20251121_apply_rbac_rls.sql`

이 문제는 가장 먼저 정리해야 한다. "역할", "앱 메타데이터", "RLS 함수", "프론트 메뉴 가드"가 하나의 권한 스키마를 공유해야 한다.

### 4-2. 프론트가 권한을 추정해서 부여함

로그인 사용자의 `app_metadata.permissions`가 비어 있으면 프론트가 자동으로 운영자 권한을 부여한다. DB가 막아준다고 해도, 화면 노출과 사용자 기대는 틀어지고, 추후 권한 정책이 바뀔 때 보안 사고로 이어질 여지가 있다.

### 4-3. 정책 로직 분산

배송비와 무료배송 기준은 이미 `site_settings`가 있지만, 실제 계산 로직은 주문 생성, 주문 수정, 주문 묶음 처리, UI 계산 등 여러 군데에 남아 있다. 지금은 작은 시스템이라 버티지만, 정책 변경이 생기면 회귀 위험이 빠르게 커진다.

### 4-4. Edge Function 응답 표준 불일치

`invite-user`는 정상이지만, `list-users`를 포함한 여러 함수가 실패에도 `200`을 반환한다. 이는 관측 가능성을 망치고, 프론트가 에러를 성공으로 오해하게 만들며, 운영 로그 분석도 어렵게 한다.

### 4-5. 테스트 신뢰도 부족

현재 테스트는 두 종류의 문제를 드러낸다.

- 실제 UI와 맞지 않는 오래된 테스트가 있다. 예: `OrderDetailModal` 테스트는 더 이상 존재하지 않는 이메일 필드를 기대한다.
- 다른 두 스위트는 `EMFILE`로 깨진다. 즉, 테스트 인프라가 현재 컴포넌트 복잡도를 안정적으로 감당하지 못한다.

## 5. 코드 수준 관찰

- `create-order`는 서버에서 가격 재계산을 수행하므로 핵심 주문 무결성은 괜찮다.
- `unlinkOrders`에는 `final_paymentValue` 같은 오타성 필드가 남아 있다. 동작에는 직접 영향이 없더라도, 이런 흔적은 유지보수 신뢰를 낮춘다.
- `fetchAllProducts` 패턴이 여러 화면에서 반복되며, 상품 수가 커질수록 대시보드/관리자 초기 로딩에 부담이 된다.
- 대시보드는 현재 "운영용 집계 화면"으로는 충분하지만, 데이터 접근과 계산 로직이 대부분 프론트에 몰려 있어 장기적으로는 성능보다 정확성 관리가 더 어려워질 가능성이 높다.

## 6. 우선순위 제안

1. 권한 체계 통합
역할/권한 문자열/RLS/프론트 가드를 한 번에 맞춰야 한다. 이게 끝나기 전에는 어떤 관리자 기능 확장도 불안정하다.

2. 정책 계산 단일화
배송비, 무료배송, 합배송 재계산을 DB 함수 또는 Edge Function 한 축으로 모아야 한다.

3. Edge Function 응답 규약 정리
실패 시 4xx/5xx를 사용하고, 프론트는 이 규약에 맞춰 에러를 처리해야 한다.

4. 테스트 정상화
깨진 기대값을 고치고, `EMFILE` 원인을 줄이도록 테스트 범위 분리 또는 설정 조정을 해야 한다.

5. 상품/대시보드 데이터 접근 최적화
전량 조회 기반 구조를 줄이고, 필요한 범위만 가져오는 쪽으로 바꾸는 것이 다음 단계다.

## 7. 결론

이 제품은 "지금 바로 현장에서 쓸 수 있는 운영 도구"라는 목적에는 이미 상당 부분 도달했다. 하지만 다음 성장 단계의 병목은 기능 부족이 아니라 권한 체계와 정책 소스의 불일치다. 지금 가장 ROI가 높은 일은 새 기능 추가가 아니라, 권한/정책/응답 규약을 정리해 시스템의 중심축을 고정하는 것이다. 그 작업이 끝나면 이후의 POS, 멀티테넌시, 고급 대시보드, 자동화 기능은 훨씬 안전하게 붙일 수 있다.
