## 🔴 우선순위 높음

### ✅ [연계 주문] parent_order_id — 추가 구매 플로우 (완료)
**배경:**
학회 현장에서 고객이 1차 결제 후 추가 구매를 원하는 경우가 빈번하게 발생함.
현재는 각 주문이 독립적으로 생성되어 출고 시 수동으로 묶어야 하는 번거로움 존재.

**구현 완료:**
- `orders.parent_order_id bigint FK` 추가
- 어드민 주문 상세 모달 → "연계 주문 연결" 버튼 → 고객명/연락처 검색 → 클릭 한 번으로 연계
- 주문 관리 목록: `#50-2` (parent), `#51(50)` (child) 포맷
- 출고 현황: parent 카드만 노출, mergedItems로 합산 표시
- 주문 상세 모달: 1차/2차 상품 목록 분리 표시, 1차/2차 결제 정보 각각 표시
- linkOrders 배송비 로직 수정 완료 (`002826e`): 합산 금액 무관하게 A가 배송비 낸 경우 항상 B에서 차감

## 🟡 중간 우선순위

### ✅ [주문 조회] 고객용 주문 상태 페이지 (구현 완료)
**배경:**
출고 알림톡 CTA가 연결될 페이지. 로그인 없이 토큰으로 접근하는 읽기 전용 페이지.

**기획 확정 내용:**

**보안 모델: 토큰 기반 URL**
- 이름+연락처 조회 폼은 만들지 않음 (같은 연구실 등 지인이 타인 정보를 알 수 있어 보안 취약)
- 토큰을 가진 사람(이메일 수신자)만 접근 가능
- 토큰 분실 시 → 어드민에서 "이메일 재발송" 버튼으로 해결

**DB 변경 필요:**
```sql
ALTER TABLE orders ADD COLUMN access_token uuid DEFAULT gen_random_uuid();
CREATE UNIQUE INDEX ON orders(access_token);
```

**라우트:** `/order/status/:token`

**표시 정보:**
- 주문 번호, 날짜
- 상태 타임라인 (주문접수 → 결제완료 → 배송중 → 수령완료)
- 합배송 안내 배너 (연계 주문인 경우): "주문 #49와 합배송, 배송비 3,000원 면제"
- 주문 상품 목록 (품목명, 수량, 할인가, 합계)
- 결제 내역 (정가 합계 / 할인 / 배송비 / 총 결제금액)
- 배송지
- 문의 이메일

**연계 주문 표시 원칙:**
- 1차 주문자(A): 자신의 주문 정보만 표시. 2차 주문자 정보 노출 안 함 (프라이버시)
- 2차 주문자(B): 자신의 주문 + "○○ 주소로 합배송" 안내

**UI 미리보기:** `/order/status/preview` (목업 라우트, 배포 전 제거)
- 단독 주문 / 연계 주문 토글 포함
- 파일: `src/components/OrderStatusPreview.jsx`

**참고 레퍼런스:** Shopify 주문 상태 페이지 (`/orders/{token}`) — 업계 표준 UX

### [알림톡] 출고 완료 카카오 알림톡 (이메일 대신)
**배경:**
의료 학회·병원 기관 이메일은 스팸 필터가 강해 도달율 불확실.
카카오 알림톡이 국내 B2C 트랜잭션 알림 표준 (오픈율 90%+).

**기획 확정 내용:**

트리거: `status → shipped` 변경 시
수신: 고객 연락처 (카카오)
내용: 발송 안내 + 주문 조회 페이지 CTA 링크 (`/order/status/:token`)
운송장 번호: ❌ 사용 안 함

**구현 필요:**
- 카카오 비즈니스 채널 등록 + 알림톡 API 연동 (카카오 developers 또는 솔루션 대행사)
- 어드민 주문 상세 모달에 "알림톡 재발송" 버튼 추가 (연락처 오탈자, 수신 실패 대응)

**구현 순서:**
1. 카카오 비즈 채널 등록 선행 (시간이 걸릴 수 있음)
2. 주문 조회 페이지 완성 후 CTA 연결 ← 주문 조회 페이지는 이미 구현됨
3. 알림톡 API 연동

- 연계 주문도 동일한 학회 설정 사용

**DB 변경 필요:**
```sql
ALTER TABLE events ADD COLUMN estimated_delivery_date date;
```

**UI 변경:**
- 어드민 학회 관리: 날짜 필드 추가
- OrderStatusPage: `events.estimated_delivery_date` 렌더링 이미 준비됨 (조건부 표시)

## 🔒 보안 이슈 (미결)

### [주문 조회] 개인정보 노출 취약점

**현황:**
- `/order/status/:id` — ✅ **해결됨 (2026-04-06):** UUID access_token으로 교체, 열거 공격 불가
- `/order/lookup` — 이름+연락처를 아는 사람이면 주소까지 열람 가능
- 같은 학회·같은 기관 동료 수준의 정보만 있어도 타인 주소 노출 위험

**가장 민감한 데이터:** 배송지 주소

**⚠️ 미결: RLS anon SELECT 정책 과잉 허용 (2026-04-06 발견)**
- `orders` 테이블에 `USING (true)` RLS 정책이 적용되어 있어, anon key를 가진 누구든
  Supabase REST API를 직접 호출하면 UUID 필터 없이 전체 주문 데이터를 조회 가능
- `curl 'https://[project].supabase.co/rest/v1/orders?select=*' -H 'apikey: [anon_key]'`
- anon key는 프론트엔드 JS 번들에 공개되어 있음
- **근본 해결:** `SECURITY DEFINER` DB function + anon SELECT RLS 삭제
  ```sql
  CREATE FUNCTION get_order_by_token(p_token uuid) RETURNS json
  LANGUAGE sql SECURITY DEFINER AS $$
    SELECT row_to_json(o) FROM (
      SELECT o.*, json_agg(oi) FROM orders o
      JOIN order_items oi ON oi.order_id = o.id
      WHERE o.access_token = p_token GROUP BY o.id
    ) o;
  $$;
  ```
- **현재 판단:** 연 800건 소규모 + UUID 난수 보호로 실질 위험 낮음 → 이메일 알림 구현 후 함께 처리

**보류 이유:**
- 이메일 없이 토큰을 전달할 수단 필요 (카카오 알림톡, SMS 등)

**후보 해결책:**
- A) 주문 상세 진입 시 이름+연락처 재확인 (프론트엔드만, 즉시 가능)
- B) 토큰 기반 URL + 이메일/알림톡으로 발송 + DB function으로 API 직접 접근 차단 (근본 해결)
- C) 주문 상세 화면에서 배송지 주소만 숨기기 (최소 조치, 즉시 가능)

**다음 결정 시 고려사항:** 이메일을 완전히 배제할 경우 토큰 전달 수단 확보가 선행 조건.
