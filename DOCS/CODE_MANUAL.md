# 📖 Inpsyt Order V2: 기획자(CTO)를 위한 코드 리터러시 매뉴얼

이 매뉴얼은 비개발자가 코드의 동작 원리와 데이터 흐름을 완벽히 이해하고 시스템에 대한 통제력을 갖추기 위해 작성되었습니다.

---

## 1. 🛒 OrderPage.jsx (주문 메인 페이지)

이 파일은 고객이 학회 링크를 타고 들어와 상품을 담고 주문을 완료하기까지의 **모든 여정(데이터 덩어리)을 들고 다니는 장바구니 카트이자 홀 매니저**입니다.

### 📌 1-1. 전체 데이터 흐름 (Data Flow)

**[1단계: 식당 문 열기 전 준비 (화면 로드)]**
1. 사용자가 특정 링크(`?events=학회슬러그`)로 접속합니다.
2. `useEffect` 라는 감시 카메라가 켜지면서, 링크에 있는 `학회슬러그`를 읽어들입니다.
3. Supabase(주방) 창고에서 "해당 학회가 오늘 열려있는지, 할인율은 얼마인지" 데이터를 가져옵니다. (`eventInfo` 저장)
4. 문제가 있으면 에러 화면(접근 차단)을 보여주고, 정상이면 주문 화면을 엽니다.

**[2단계: 손님의 주문 활동 (데이터 누적)]**
1. **상품 선택 (`activeStep === 0`)**: 손님이 `ProductSelectionStep`에서 상품을 고르면, 그 정보가 `cart`라는 변수에 차곡차곡 쌓입니다.
2. **정보 입력 (`activeStep === 1`)**: 손님이 `CustomerInfoStep`에서 이름/연락처 등을 적으면, `customerInfo`라는 변수에 저장됩니다.
3. **최종 확인 (`activeStep === 2`)**: 여태까지 모은 `cart`, `customerInfo`, 할인율 등을 합쳐서 화면에 영수증(`totalPrice`)으로 뿌려줍니다.

### 📌 1-2. 결정적 순간: "결제(주문 접수) 버튼"을 눌렀을 때의 동선

화면 맨 밑의 `FloatingBottomBar`에서 **[주문 접수하기] 버튼**을 눌렀을 때의 내부 동선입니다.

1. **[클릭]** 사용자가 버튼을 누름.
2. **[검사]** `OrderPage.jsx`의 `handleSubmitOrder` 함수가 호출됨. 
   - *로직:* "학회 정보가 잘 있나? 없으면 에러메시지!"
3. **[포장]** 지금까지 들고 있던 `customerInfo`(고객정보)와 `cart`(장바구니) 데이터를 하나의 큰 상자(`body`)로 예쁘게 포장합니다.
4. **[무전 발송]** `supabase.functions.invoke('create-order')` 코드가 실행됩니다.
   - *의미:* 홀 매니저가 주방(백엔드 Edge Function)으로 데이터 상자를 휙 던지면서 "주문 들어왔어요!"라고 무전기를 칩니다.
5. **[주방 작업 (DB 도달)]** 주방(Edge Function)에서 DB의 `orders`, `order_items` 테이블에 데이터를 써놓고, 이메일까지 발송합니다. ==**이메일 발송 안하고 있음**==
6. **[결과 수신]** 주방에서 "완료!"라는 응답을 프론트엔드로 다시 넘겨줍니다.
7. **[마무리]** 프론트엔드는 응답을 받아 `showSuccessDialog` 스위치를 켜서 손님에게 **"주문이 접수되었습니다" 팝업**을 띄웁니다.

### 📌 1-3. 주요 함수 (Functions) 해부 테이블

어떤 재료(Input)를 넣으면 무슨 동작(Output)을 하는지 요약한 표입니다.

| 함수 이름 | 입력값 (Input) | 출력/결과 (Output) | 역할 (웨이터의 행동) |
| :--- | :--- | :--- | :--- |
| **`fetchEventData`** | URL의 `eventSlug` | `eventInfo` (학회 정보) 또는 에러 상태 | 손님이 들고 온 초대장(주소)을 보고, 유효한 학회인지 DB를 조회해 기억함. |

| **`handleSubmitOrder`** | (화면에 입력해둔) `cart`, `customerInfo` | 주문 성공 팝업 (`showSuccessDialog = true`) | 손님이 적어준 주문표를 주방(DB)으로 넘기고, 성공/실패 여부를 알려줌. |
| **`handleCloseSuccessDialog`** | 없음 | `cart`, `customerInfo`, `activeStep` 모두 초기화 | 결제가 끝난 후 테이블을 싹 치우고 다음 손님을 받을 준비를 함. (초기화) |

## 2. 👨‍🍳 supabase/functions/create-order/index.ts (주문 생성 백엔드 함수)

이 파일은 `OrderPage.jsx`에서 날아온 "주문 들어왔습니다!"라는 무전을 받고, **실제로 데이터를 안전하게 처리해 창고(DB)에 적어넣는 핵심 주방(백엔드 로직)**입니다.

### 📌 2-1. 전체 데이터 흐름 (Data Flow)

**[1단계: 주문서 수령 및 검증]**
1. 프론트엔드에서 보낸 데이터(이름, 연락처, 장바구니 품목들, 학회 ID)를 넘겨받습니다. `req.json()`
2. **[보안 검증 🛡️]** 프론트엔드에서 계산한 "총 금액"을 절대 믿지 않습니다! (누군가 웹브라우저에서 값을 조작해 10원만 결제할 수도 있으니까요).
3. 장바구니에 담긴 상품 ID(`product_id`)와 학회 ID(`event_id`)를 가지고 **DB 창고를 직접 뒤져서 진짜 정가와 할인율**을 가져옵니다.

<!-- ==**Q. 검증이 틀렸다면 어떻게 하는지, 현재 코드에서 다르게 조작하는 방법이 있는지 검토**== -->

**[2단계: 요리 및 계산 (서버사이드 조작)]**
1. DB에서 가져온 진짜 정가와 할인율을 바탕으로 **원래 가격(`totalOriginalPrice`), 할인가격(`totalDiscountedPrice`), 배송비(`shippingCost`)**를 주방 안에서 아주 안전하게 다시 계산합니다.
2. 최종 결제 금액(`finalCost`)을 확정합니다.

**[3단계: 창고(DB)에 영구 기록]**
1. 확정된 데이터를 `orders` 테이블(총괄 주문서)에 새 줄(Row)로 적어 넣습니다. 
2. 방금 만들어진 주문서의 번호(`newOrder.id`)를 따와서, `order_items` 테이블(개별 상품 내역)에 하나씩 꼼꼼하게 기록합니다.

**[4단계: 알림 및 서빙 (마무리)]**
1. **[이메일 발송]** 이메일 주소가 있다면, Resend API를 통해 "주문 접수 완료" 이메일을 발송합니다. (이때 메일 발송이 실패해도 주문 기록은 정상적으로 완료되게끔 독립적으로 처리합니다: Fire-and-forget). 
   <!-- ==**이메일 발송 안하고 있음**== -->
2. 프론트엔드 홀 매니저에게 "성공적으로 주문 접수했다. 이 영수증 번호(`newOrder.id`)를 손님한테 줘라!"라며 200(성공) 신호를 보냅니다.

### 📌 2-2. 주요 함수/로직 해부 테이블

| 로직 블록 (동작) | 입력값 (Input) | 출력/결과 (Output) | 역할 (주방의 행동) |
| :--- | :--- | :--- | :--- |
| **`req.json()` 파싱** | 프론트엔드가 보낸 `body` 객체 | 이름, 연락처, `cart`, `event_id` 등 변수 | 홀 매니저가 들고 온 주문서를 받아쓰는 작업. |
| **DB 조회 (`supabaseClient.from...`)** | `cart`의 상품 ID들, `event_id` | `products`(가격표), `event`(할인율) 데이터 | 주방 벽에 붙어있는 공식 가격표와 행사 할인표를 눈으로 확인하는 작업. |
| **가격 재계산 (Recalculate)** | `products`, `event`, `cart`(수량) | `finalCost`, `shippingCost` 등 찐 결제금액 | 주방장이 직접 안전띠를 두르고 계산기 두드리는 작업. (보안 핵심!) |
| **DB 저장 (`insert`)** | 최종 계산된 데이터 꾸러미 | 생성된 `newOrder` 정보 | "다 만들었으니 장부에 펜으로 확정해서 적자!" |
| **이메일 발송 (`fetch(Resend)`)** | 고객 이름, 이메일, `finalCost` | 이메일 발송 완료 (결과와 무관 진행) | 우체통에 편지를 넣는 작업. (넣고 나서 편지가 잘 갔는지까지 기다리진 않음) |

## 3. 📋 ProductSelectionStep.jsx & CustomerInfoStep.jsx (프론트엔드 - 메뉴판과 주문서)

이 두 파일은 `OrderPage.jsx`(홀 매니저)가 손님에게 보여주는 **[1단계: 메뉴판]**과 **[2단계: 개인정보 입력지]**입니다. 자신이 맡은 화면만 그리고, 사용자가 입력한 값은 부모(`OrderPage`)에게 전달하는 철저한 분업을 하고 있습니다.

### 📌 3-1. ProductSelectionStep.jsx (상품 선택 화면) 전체 흐름

**[1단계: 진열대 채우기 (데이터 로드)]**
1. 컴포넌트가 화면에 뜨자마자(`useEffect`), 백엔드에서 판매할 상품 목록을 가져옵니다. (`loadProducts`)
2. 이때 부모가 알려준 현재 학회(`eventTags`)에 해당하는 상품만 진열대에 올립니다.

**[2단계: 손님의 탐색 (스마트 검색 및 필터)]**
1. 손님이 검색창에 글자를 치거나 카테고리(인기, 도서, 검사) 칩을 누릅니다.
2. 매번 키보드를 칠 때마다 서버에 무전을 보내면 서버가 터지므로, **0.3초(300ms) 동안 입력이 멈추면 그때 한 번만 묶어서 API를 호출**합니다. (이를 프로그래밍 용어로 Debounce 기법이라고 합니다).

<!-- ==(단순궁금)lazy loading과 다른거지?== -->


**[3단계: 장바구니 조작]**
1. 손님이 [+추가], [+], [-] 버튼을 누릅니다. (`handleAddProduct`, `handleIncrement` 등)
2. 이 컴포넌트는 독단적으로 장바구니를 수정하지 않고, 부모(`OrderPage`)가 건네준 무전기(`onCartChange`)를 통해 "매니저님! 카트 수량 이렇게 바꿔주세요!"라고 요쳥만 합니다.

### 📌 3-2. CustomerInfoStep.jsx (고객 정보 입력 화면) 전체 흐름

**[1단계: 입력란 제공 및 부모에게 전달]**
1. 텅 빈 양식을 띄웁니다. 사용자가 글씨를 지웠다 쓸 때마다 부모의 무전기(`setCustomerInfo`)를 써서 부모 장부를 실시간 갱신합니다.

**[2단계: 편의 기능 작동 (스마트 UX)]**
기획자로서 눈여겨볼 만한 "Vibe Coding이 꽤 친절하게 짜놓은 UX 로직"들입니다.
1. **전화번호 자동 하이픈:** 손님이 `0101234`까지만 쳐도 코드가 알아서 `010-1234`로 숫자를 잘라 띄워줍니다. (`handlePhoneChange`)
2. **이메일 자동완성:** 'test'만 치면 자동으로 아래에 `@naver.com`, `@gmail.com` 추천 목록이 드롭다운으로 뜹니다. (`emailSuggestions` 변수)
3. **주소 검색:** '주소 검색' 버튼을 누르면 다음(Daum) 우편번호 웹을 모달로 띄웁니다. 선택이 완료되면 우편번호와 기본 주소를 쪼개서 우리 양식에 자동으로 꽂아줍니다. (`handleCompletePostcode`)

### 🛑 [CTO's Checkpoint] (나중에 수정할 부채)
- **과도한 서버 호출 (API 낭비):** `ProductSelectionStep`에서 '도서'나 '검사' 칩을 누를 때마다 굳이 서버(`fetchProducts`)를 다시 부르고 있습니다. 전체 상품 수가 수천 개가 넘지 않는 이상, 처음에 한 번 100개를 다 가져오고 브라우저 안에서만 숨겼다 보여줬다 처리하는 게 훨씬 빠르고 서버 비용을 다이어트할 수 있습니다.

## 4. 📡 api/orders.js (프론트엔드 - 무전기 및 통신 규약)

이 파일은 프론트엔드(화면)와 백엔드(DB) 사이를 이어주는 **전담 무전병(API 통신 모듈)**입니다.
여기서는 손님(OrderPage)보다는 주로 **사장님(Admin)이 매장을 관리할 때** 사용하는 크고 무거운 데이터 요청들을 전담합니다.

### 📌 4-1. 주요 무전 채널(함수) 해부

이 파일에 정의된 함수들은 화면을 직접 그리지 않습니다. 오직 \"DB에 가서 이것 좀 가져와!\", \"DB에 가서 이것 좀 바꿔줘!\"라고 데이터 배달부 역할만 합니다.

| 함수 이름 | 입력값 (Input) | 출력/결과 (Output) | 역할 (무전병의 임무) |
| :--- | :--- | :--- | :--- |
| **`getOrders`** | 페이지 번호, 검색어, 학회 종류 | 전체 주문 목록 (페이지네이션 적용) | \"사장님, 여기 검색하신 조건에 맞는 주문 장부 가져왔습니다.\" |
| **`getFulfillmentOrders`** | 학회 종류, 날짜 구간 | 주문 내역 + 포함된 상품명 상세 | 출고팀을 위해 주문서와 그 안에 담긴 상품 이름까지 한 번에 묶어서(Join) 가져옴. |
| **`linkOrders`** | 부모 주문 번호, 자식 주문 번호 | 배송비가 재계산된 결과 | \"합배송 처리할게요! 두 개 합쳐서 3만 원 넘으면 배송비 빼고 DB 시스템 업데이트 하겠습니다.\" |
| **`searchOrdersForLinking`** | 전화번호나 이름 검색어 | 합배송 묶을 수 있는 주문 후보들 | \"이름이나 전화번호가 비슷한 주문들 찾아왔습니다. 부모 주문으로 묶으시겠어요?\" |

### 📌 4-2. 기획자가 알아야 할 '합배송(링크)' 로직의 비밀

`linkOrders` 함수 안에는 꽤 복잡한 비즈니스 로직이 들어있습니다.
1. A 주문과 B 주문의 정가(\`total_cost\`)를 합칩니다.
2. 합친 금액이 **무료배송 기준(30,000원)**을 넘는지 검사합니다.
3. 만약 넘는다면, 그리고 A 주문(부모)이 이미 배송비를 냈다면? → B 주문(자식)의 최종 결제 금액에서 그 배송비만큼을 아예 차감해버려 손님의 손해를 막습니다.
4. 만약 안 넘는다면? → B 주문(자식)의 배송비만 0원으로 만들어서 합배송 시킵니다.

### 📌 5-1. AdminLayout이 하는 3가지 핵심 역할

사장님이 사무실에서 딴짓(?)을 하거나 다른 장부를 보고 있는 중에도, 바깥 홀에서는 계속 주문이 들어옵니다. 매번 페이지를 새로고침(F5) 할 수는 없겠죠?

## 8. 🗃️ supabase/functions/list-users & delete-user (어드민 직원 명부 관리)

이 두 파트는 사장님(master)이나 인사담당자가 **직원 명부를 조회하고(list) 해고하는(delete) 기능**을 수행합니다.

### 📌 8-1. list-users (직원 조회)의 이중 장부 구조

Supabase는 무서운 해킹을 방지하기 위해 **'진짜 가입 정보(비밀번호 등)'**와 **'화면에 보여줄 프로필 정보(이름, 부서)'**를 두 개의 장부로 엄격히 분리해서 보관합니다.
1. `supabaseAdmin.auth.admin.listUsers()`: 굳게 닫힌 비밀 금고(Auth)에서 등록된 계정(이메일, 권한)들을 빼옵니다.
2. `supabaseAdmin.from('user_profiles')`: 공개 캐비닛(DB)에서 사원들의 이름표와 직책을 빼옵니다.
3. 이 두 장부를 함수 안에서 하나의 엑셀표(Map)처럼 재조립해서, 최종적으로 프론트엔드 어드민 화면에 뿌려줍니다.

*(기획자님께서 아까 메모하신 "DB 동기화가 진짜인가? 확인필요"에 대한 답변입니다. 네, 동기화가 확실히 이루어집니다! 7번에서 사원증을 발급할 때 이 두 장부에 동시에 다 적어두기 때문에 여기서 이렇게 합쳐서 조회가 가능한 것입니다.)*

### 📌 8-2. delete-user (직원 해고)의 단호함

1. 오로지 **`master` 직급을 가진 진짜 사장님만** 찌를 수 있는 백엔드 API입니다. (직원을 추가할 수 있는 인사담당자라도 직원을 지울 수는 없도록 하드코딩 되어 있습니다).
2. 직원을 지우면 비밀 금고(Auth)에서만 지우는 게 아니라, DB(public.user_profiles)에 있는 이름표도 연쇄 폭발(Cascade 적용)로 같이 지워지도록 설계되어 있습니다.

### 🛑 [CTO's Checkpoint] (나중에 수정할 부채)
- **에러 처리의 동일한 문제점:** 여기서도 여전히 에러가 나면 200 성공 코드를 던집니다. 즉각적인 리팩토링이 필요합니다.
- **권한 매트릭스화:** 기획자님이 남겨주신 **`{페이지별 * 권한(수정, 지우기, 쓰기, 읽기)} 형태의 직관적인 관리`** 메모를 이 시스템에 그대로 도입(리팩토링)해야 합니다. 현재는 낡고 하드코딩된 역할(role) 체계에 머물러 있는데, 이를 체크박스 형태의 매트릭스(Matrix) 권한으로 바꾸면 훨씬 고도화된 ERP 시스템으로 진화합니다.

---

1. "페이지별 * 권한 매트릭스 형태로 관리하는게 더 직관적이지 않을까?" 👉 완전 정답! 지금 코드는 구시대적인 권한 부여 방식(단순 하드코딩 배열)을 쓰고 있습니다. 이걸 체크박스가 달린 **{메뉴 권한 매트릭스표}**로 바꾸는 기획이야말로 나중에 이 시스템에 '출판팀', '영업팀' 등 복잡한 부서가 추가될 때를 대비하는 가장 유연한 "상용 어드민 구축 전략"입니다.

2. "진짜인가 확인 필요 (DB 동기화)" 👉 확인 완료했습니다! 생성 성공 직후 user_profiles 데이터베이스에 바로 이름표(upsert)를 꽂아두도록 설계되어 있습니다. 이 덕분에 사장님이 나중에 전체 명부를 볼 수 있습니다.

3. "수정사항 전 지명반론자법, 3Whys 탐색" 👉 크... 이건 정말 개발팀을 이끄는 완벽한 철학입니다. 무언가를 뜯어고치기 전에 **"왜 이걸 고쳐야 하지? 안 고치면 당장 망하는가? 더 쉬운 방법은 없나?"**를 묻는 훈련이 저희를 Vibe Coding의 수렁에서 건져줄 밧줄이 될 것입니다. 저도 이 철학에 똑같이 동참하겠습니다.

## 9. ⚙️ 데이터베이스 내부 톱니바퀴 (SQL Functions & Triggers)

프론트엔드나 엣지 함수(Edge Function)가 데이터를 조작하기도 전에, DB 창고 그 자체에 내장시켜버린 '자동화 로직'들입니다. 외부 인터넷이 끊기든 서버가 죽든, DB 안에서 안전하게 돌아가는 가장 견고한 처리 방식입니다.

### 📌 9-1. update_order_with_transaction.sql (트랜잭션: 올 오어 나띵)

어드민 화면에서 사장님이 '주문 내역'이나 '품목'을 수정하고 저장할 때 작동하는 전용 함수(`update_order_with_items`)입니다.
**[왜 이렇게 짰을까?]** 
주문 정보를 고치고 나서, 바뀐 품목 정보를 고치려는 찰나에 서버가 뻗으면 어떻게 될까요? 결제 금액은 5만 원으로 줄었는데, 품목은 옛날 10만 원어치가 그대로 남아 구멍이 나는 대참사가 일어납니다.
이 함수는 **"고칠 거면 한 폭의 그림처럼 동시에 다 고치든지, 중간에 삑사리 나면 아예 안 고친 걸로 되돌려 놔!" (Transaction 패턴)** 전략을 씁니다. 기존 장바구니(`order_items`) 내용을 싹 다 지운 뒤 새 내용으로 한꺼번에 밀어 넣습니다. 

### 📌 9-2. create_paid_order_trigger.sql (자동 알림 방아쇠)

말 그대로 사격장에 있는 '방아쇠(Trigger)'입니다. 프론트엔드나 엣지 함수가 "알림 보내!"라고 명령하지 않아도, DB 창고가 실시간으로 수문장처럼 지켜보다가 자기 스스로 총을 쏩니다.
- **감시 조건:** 특정 주문서(`orders` 테이블)의 상태(`status`)가 방금 전에는 '결제 전'이었는데, 업데이트를 통해 지금 막 **'결제 완료(paid)'**로 확실히 바뀌었는가?
- **격발 행동:** 조건이 맞으면 `notify_order_paid()` 함수가 실행되어 매장 전역에 `order_paid`라는 사이렌(pg_notify)을 울립니다.
- *(이 사이렌 소리를 듣고 다른 자동화 봇(Edge Function)이 깨어나 "결제완료 이메일을 발송해라!" 하고 움직이게 만드는 고도의 백엔드 아키텍처 패턴입니다.)*
