# 04. Claude 디자인 프롬프트

> 이 문서는 시안 작업에서 Claude에게 그대로 붙여 넣는 프롬프트 모음이다. 다른 문서들보다 톤이 다르다. 자연어가 아니라, 시안 도구에 직접 들어갈 기능형 텍스트다. 그래도 각 프롬프트 앞에 "이 화면이 왜 어려운가"를 한두 문장씩 붙여두었다. 그 한 줄을 시안 작업자가 머리에 넣은 채로 결과를 봐야 한다.

## 어떻게 쓰는가

각 프롬프트 블록을 통째로 복사해 시안 도구(예: /design-html, /design-shotgun)에 넣는다. 변수가 들어가는 자리는 중괄호로 표시했다. 학회 이름, 상품 카테고리 같은 건 작업 시점 데이터로 대체한다.

프롬프트는 12개다. 고객 화면 4개, 어드민 화면 7개, 공통 셸 1개.

## 공통 머리말 (모든 프롬프트의 앞에 자동으로 붙는 컨텍스트)

[프로젝트] 인싸이트 현장주문, 의료 학회 현장 도서·검사 주문 시스템
[규모] 연 8일 운영, 연 800건, 의사·연구자 개인 비용 구매, 재구매 거의 없음
[톤] 신뢰감과 부드러움. 광고 사이트 톤, 과한 그라데이션, 비비드 색 금지.
[가독성] 본문 16px 기본, 최소 14px, 12px은 캡션과 법적 고지에만.
[색 사용 규칙] 색은 의미가 있을 때만. 회색이 메인, 강조색은 의미 단위로만 등장.
[다크모드] 별도 다크 테마 없음. prefers-color-scheme 감지하여 자동 보정만.
[접근성] WCAG AA 기본, 터치 영역 44x44px 최소, 줌 150%까지 레이아웃 유지.
[디자인 원칙] design-system/01_DESIGN_PRINCIPLES.md 6개 원칙을 반드시 따른다.

## 고객 화면 프롬프트

### C1. 주문서 1단계, 상품 선택

**이 화면이 왜 어려운가.** 학회장 의자에 앉은 50대 의사가 처음 보는 화면이다. 상품 30~80종이 한 페이지에 떠 있고, 그분은 카테고리 1~2개에서 1~3개 상품만 사면 끝난다. 너무 화려하면 부담스럽고, 너무 단조로우면 어떤 상품인지 못 찾는다. 가격이 보이지 않으면 본인 돈 쓰는 사람의 손이 멎는다.

[화면] 고객 주문서, 1단계 상품 선택
[디바이스] 모바일 (390x844 기준, 태블릿 768까지 대응)
[상황] 학회장 부스 앞에서 QR로 진입. 30~80개 상품 중 1~3개 선택.

[필수 요소]
- 상단 고정: 학회 이름 + "1/3단계 상품 선택" 표시
- 검색창 (상단)
- 카테고리 칩 (검사·도서·세트, 가로 스크롤)
- 정렬 필터 (인기·신상품·할인순)
- 상품 그리드 카드: 썸네일, 상품명, 정가(작고 옅게 윗줄), 할인가(크고 진하게 아랫줄), 할인율 뱃지
- 하단 고정 장바구니 바: 담은 개수, 합계 금액, "주문서 작성" CTA, 무료배송 임계금액까지 남은 금액 안내

[원칙 반영]
- 가격 두 줄 위계 명확히 (원칙 1)
- 할인율 뱃지는 의료 톤으로 절제 (원칙 1)
- 카드 라운드 10, 그림자 한 단계만 (원칙 6)
- 본문 16px (원칙 5)

### C2. 주문서 2단계, 배송 정보 입력

**이 화면이 왜 어려운가.** 모바일 입력 폼은 누구에게나 어렵다. 50대 의사라면 더 그렇다. 인싸이트 ID(검사 구매 시), 다음 우편번호 모달, 요청사항까지 한 화면에 있다. 잘못 입력하면 정정이 안 되거나, 학회 끝나고 운영자가 전화해야 한다.

[화면] 고객 주문서, 2단계 배송 정보
[디바이스] 모바일
[상황] 1단계에서 상품 선택 완료 후 진입. 본인 정보 입력.

[필수 요소]
- 상단: "2/3단계 배송 정보" + 뒤로가기
- 입력 필드: 이름, 연락처, 인싸이트 ID(검사 구매 시 노출), 우편번호 검색 버튼, 도로명주소, 상세주소, 요청사항(선택)
- 각 필드 라벨은 input 위에 (placeholder 안에 라벨 두지 않음)
- 필수 표시는 빨간 별표 대신 "(필수)" 텍스트
- 우편번호 검색: 다음 주소 모달 호출
- 하단 고정: "다음 단계로" CTA

[원칙 반영]
- 라벨이 input 위에 위치 (원칙 5, 가독성)
- 인싸이트 ID 필드는 검사 상품 담긴 경우만 노출 (원칙 2)
- 에러 메시지는 필드 바로 아래, 왜 안 됐는지와 무엇을 하면 되는지 (원칙 4)
- 터치 영역 44x44 최소 (원칙 5)

### C3. 주문서 3단계, 최종 확인

**이 화면이 왜 어려운가.** 본인 돈 쓰는 사람이 마지막으로 멈추는 화면이다. 여기서 의심이 풀려야 결제가 일어난다. "정말 이 금액이 맞나"가 0.5초 안에 풀려야 한다.

[화면] 고객 주문서, 3단계 최종 확인 + 제출
[디바이스] 모바일

[필수 요소]
- 상단: "3/3단계 주문 확인"
- 주문 상품 요약 카드 (상품명·수량·금액, 접고 펴기 가능)
- 배송지 요약 카드 (수정 가능)
- 금액 정리: 상품 금액, 할인, 배송비(임계금액 안내 포함), 합계
- 합계는 가장 크고 진하게
- 배송 예정일 표시 (학회 종료 후 D+N 형식)
- 약관 동의 체크
- 하단 고정: "결제하러 가기" CTA (부스 직원에게 카드 건네는 흐름임을 안내하는 한 줄 텍스트 포함)

[원칙 반영]
- 합계 금액의 크기가 화면에서 가장 크다 (원칙 1, 2)
- 배송 예정일이 명확히 보인다 (원칙 1)
- "결제하러 가기"는 brand 색 CTA, 그 외 버튼은 약하게 (원칙 3)

### C4. 주문 조회 페이지

**이 화면이 왜 어려운가.** 결제 끝나고 알림톡 받은 사람이 링크 누르면 도착하는 곳. "내가 산 게 뭐였지", "언제 오지"가 답이어야 한다. 우리는 출고 알림이 없기 때문에 이 페이지가 사실상 유일한 "상태 확인 창구"다.

[화면] 고객 주문 조회 페이지
[디바이스] 모바일·태블릿·PC 다 대응
[상황] 알림톡 버튼 또는 QR로 진입. UUID access_token 기반.

[필수 요소]
- 상단: 학회 이름 + 주문 번호 (작게)
- 주문 상태 큰 라벨 (결제완료·배송준비·배송완료 등)
- 배송 예정일, 큰 글씨로 강조
- 주문 상품 목록 (썸네일·상품명·수량·금액)
- 배송지 요약
- 결제 정보 (할인 적용 후 금액)
- 문의 안내 (인싸이트 채널 또는 부스 연락처)

[원칙 반영]
- 주문 상태와 배송 예정일이 화면 상단에 가장 크게 (원칙 2)
- 상태 색은 토큰 status- 카테고리만 사용 (원칙 6)
- "출고 알림이 없습니다" 같은 안내는 명시하지 않음. 대신 "배송 예정일 기준 학회 종료 후 일괄 발송됩니다" 한 줄

## 어드민 화면 프롬프트

### A1. 어드민 주문 관리, 리스트

**이 화면이 왜 어려운가.** 부스 운영자가 가장 자주 보는 화면이다. 100건 중 99건은 여기서 끝난다. 필터·검색·체크박스 다중 선택·일괄 상태 변경·페이지네이션이 한 화면에 다 있어야 하면서, 시끄러운 부스에서 잘못 누르지 않아야 한다.

[화면] 어드민 주문 관리 페이지, 리스트 뷰
[디바이스] 태블릿 세로 (820x1180 기준), 데스크탑 대응
[상황] 학회 현장 운영자가 100건/일 처리. 검색·필터·일괄 상태 변경이 핵심.

[필수 요소]
- 상단 헤더: "주문 관리" 제목, 학회 필터(멀티), 신규 주문 추가 버튼, 엑셀 다운로드 버튼
- 검색바: 고객명 검색
- 필터 행: 주문 상태 멀티 칩, 날짜 범위
- 선택된 주문이 있을 때만 나타나는 액션 바: "N건 선택" + 일괄 상태 변경 드롭다운
- 테이블: 체크박스, 주문번호(짧게), 학회, 고객명, 연락처 일부, 상품 요약, 합계, 상태 칩, 결제일시
- 행 클릭 시 상세 모달 열기
- 페이지네이션 (20건/페이지 기본)

[원칙 반영]
- 자주 하는 일은 빠르게. 체크박스 후 상단 일괄 변경, 두 탭 (원칙 3)
- 상태 칩 색은 의미 단위, 페이지 전체에서 통일 (원칙 6)
- 행 높이는 손가락 잘못 닿지 않도록 56px 이상 (원칙 5)
- 빈 상태: "검색 결과가 없습니다" + "신규 주문 추가" 버튼 (원칙 4)

### A2. 어드민 주문 상세 드로어

**이 화면이 왜 어려운가.** 학회장에서 일어나는 99%의 정정이 여기서 해결되어야 한다. 모든 필드가 편집 가능해야 하면서도, 취소·환불처럼 되돌릴 수 없는 동작은 한 박자 늦춰야 한다. 일반 모달보다 드로어가 더 적합한 이유는 리스트와 상세를 동시에 보면서 정정하는 운영자 흐름이 많기 때문이다.

[화면] 어드민 주문 상세 드로어
[디바이스] 태블릿 세로 (오른쪽에서 슬라이드 인, 가로 480~520px)
[상황] 운영자가 리스트에서 한 건 클릭, 정정 또는 상태 변경.

[필수 요소]
- 상단: 닫기, 주문번호, 결제 상태 칩
- 고객 정보 섹션 (이름·연락처·인싸이트 ID·배송지), 모든 필드 편집 가능
- 상품 항목 섹션, 수량 수정, 항목 추가·삭제
- 금액 정리 섹션, 자동 계산
- 합배송(연계 주문) 섹션, 연결·해제 버튼
- 상태 변경, 드롭다운, 변경 시 확인 한 번
- 알림톡 재발송 버튼, 확인 모달 한 번
- 하단 고정: 저장 / 닫기. 취소·환불은 별도 위험 액션 버튼으로 분리, 색·위치 다르게

[원칙 반영]
- 모든 필드 편집 가능 (원칙 4)
- 위험 액션은 색·위치 분리 + 확인 모달 (원칙 3)
- 상태 칩과 결제 상태가 한눈에 (원칙 2)
- 인라인 에러 메시지로 "왜 안 됐는지 + 어떻게 하면 되는지" (원칙 4)

### A3. 어드민 출고 현황

**이 화면이 왜 어려운가.** 학회 끝난 뒤 출고팀이 본다. 결제완료 주문만 추리고, 도서·검사 분리, 합배송 그룹 묶기. 이름·연락처·인싸이트 ID·배송지를 클릭 한 번에 복사해서 엑셀이나 송장 시스템에 붙여 넣는다. 클릭 복사가 안 되면 시간이 두 배로 든다.

[화면] 어드민 출고 현황 페이지
[디바이스] 데스크탑·태블릿 가로

[필수 요소]
- 상단: "출고 현황" 제목, 행사 필터, 도서·검사·전체 토글
- 결제완료 단계(paid·preparing·completed) 주문만 표시
- 테이블 또는 카드 리스트: 주문번호, 고객명, 연락처, 인싸이트 ID, 배송지(우편번호 + 도로명 + 상세 분리 복사)
- 각 필드 우측에 복사 아이콘. 클릭 시 토스트로 "복사됨" 알림
- 합배송 그룹은 시각적으로 묶음 표시 (배경색 또는 좌측 인디케이터)
- 출고 상태 변경 버튼

[원칙 반영]
- 클릭 복사 가능 영역은 호버·탭 시 명확히 강조 (원칙 4)
- 토스트는 5초 머문다 (원칙 4)
- 합배송 그룹은 한눈에 보이게 (원칙 2)
- 배송지 우편번호 표기하되 복사 시 우편번호 제외 (원칙 4)

### A4. 어드민 대시보드

**이 화면이 왜 어려운가.** 매출 KPI·YoY·순위·실시간 알림이 한 화면에 들어와야 하면서도, 매일 같은 사람이 보는 화면이 아니다. 학회 첫날엔 한 시간에 두세 번 보고, 그 외 날엔 거의 안 본다. 그러니까 "오늘 어떻다"가 1초에 잡혀야 한다.

[화면] 어드민 대시보드
[디바이스] 데스크탑·태블릿 가로

[필수 요소]
- 상단: 연도·학회·상세 행사 3단 필터
- KPI 카드 4종 가로 배치: 검사 판매액, 도서 판매액, 배송비, 총 매출액
- 각 KPI 카드에 YoY 증감 표시 (위·아래 화살표 + 퍼센트)
- 오늘 주문 수 + 상태 분포 바
- 검사 Top5 / 도서 Top5 표 또는 리스트
- 최근 주문 5건, 클릭 시 상세 모달
- 우상단 실시간 알림 토스트 (신규 주문 INSERT 시)

[원칙 반영]
- KPI는 숫자가 가장 크다 (원칙 2)
- YoY 화살표 색은 status- 토큰 (긍정·부정) 일관 (원칙 6)
- 빈 상태: "선택한 기간에 데이터가 없습니다" (원칙 4)
- 차트는 색 최소화, 회색 + brand 한 종 위주 (원칙 1, 2)

### A5. 어드민 학회 관리

**이 화면이 왜 어려운가.** 학회·행사 CRUD에 QR 생성·URL 복사가 붙어 있다. 학회 직전 며칠 동안만 자주 만지고 그 외엔 안 만진다. "지금 진행 중인 학회는 어떤 거지" 하나가 빠르게 풀려야 한다.

[화면] 어드민 학회 관리 페이지

[필수 요소]
- 탭: 학회(societies) / 행사(events)
- 행사 리스트: 이름, 슬러그, 기간, 상태 칩(예정·진행중·종료), 주최 학회, 시즌, 액션
- 상태 자동 분류, 색 통일 (status-active·upcoming·ended)
- 행사 액션: 주문 URL 새 창, URL 복사, QR SVG 다운로드, 편집
- 신규 행사 추가 모달: 이름·슬러그·기간·할인율·배송 예정일

[원칙 반영]
- "진행중" 상태는 시각적으로 가장 강조 (원칙 2)
- QR·URL 복사는 클릭 한 번에 토스트 피드백 (원칙 4)
- 행사 삭제는 위험 액션으로 분리 (원칙 3)

### A6. 어드민 상품 관리

**이 화면이 왜 어려운가.** 상품 수백 개. 엑셀 업로드, 다중 선택, 일괄 편집, 전체삭제(받아쓰기)까지 들어 있다. 운영자가 가장 무서워하는 화면 중 하나라 위험 액션의 차단이 핵심이다.

[화면] 어드민 상품 관리 페이지

[필수 요소]
- 상단: 검색, 카테고리 필터, 신규 상품 / 엑셀 업로드 버튼
- 체크박스 다중 선택 + 선택 액션 바: "N건 선택" + 일괄 편집 / 선택 삭제
- 엑셀 업로드 진행 모달: 진행률 바, 청크별 상태, 행 단위 오류 보고
- 전체삭제 버튼은 별도 위험 영역, 받아쓰기 확인 ("전체삭제" 타이핑)

[원칙 반영]
- 위험 액션은 위치·색 분리 + 받아쓰기 (원칙 3)
- 엑셀 업로드 오류는 행 단위로 어떤 행이 왜 안 됐는지 (원칙 4)
- 빈 상태: 엑셀 업로드 안내 (원칙 4)

### A7. 어드민 사용자 관리 + 역할 템플릿

**이 화면이 왜 어려운가.** 매트릭스 UI로 역할별 권한을 편집한다. 잘못 만지면 운영 권한이 흔들리는 화면이다. 자주 안 만지지만, 만질 때 분명히 보여야 한다.

[화면] 어드민 사용자 관리 페이지

[필수 요소]
- 탭 1, 사용자 목록: 이름, 역할, 마지막 로그인, 액션(권한 부여·해제, 비밀번호 초기화, 삭제)
- 탭 2, 역할 템플릿: 매트릭스 UI (행: 역할, 열: 권한, 셀: 체크박스)
- 신규 역할 생성, 기존 역할 편집·삭제
- master 권한이 있는 사용자는 시각적으로 구분

[원칙 반영]
- 매트릭스 셀은 충분히 크게 (원칙 5, 터치 영역)
- 권한 변경 저장 시 확인 모달 한 번 (원칙 3)
- master 권한 부여·해제는 별도 위험 액션 (원칙 3)

## 공통 셸 프롬프트

### S1. 어드민 공통 셸 (헤더 + 사이드 네비)

**이 화면이 왜 어려운가.** 모든 어드민 페이지가 이 셸 안에 들어간다. 셸이 흔들리면 페이지 사이 일관성이 무너진다. 게시판 읽지 않음 뱃지, 알림 아이콘, 피드백 제출 등이 헤더에 산다.

[컴포넌트] 어드민 공통 셸, 헤더 + 사이드 네비
[디바이스] 태블릿 세로 + 데스크탑

[필수 요소]
- 상단 헤더: 로고 + 현재 페이지 제목, 검색(전역, 옵션), 피드백 제출 버튼, 알림 아이콘(세션 알림), 사용자 메뉴(이름·로그아웃)
- 좌측 사이드 네비: 대시보드, 주문 관리, 학회 관리, 상품 관리, 출고 현황, 사용자 관리, 게시판, 설정, 피드백 관리(master)
- 사이드 네비는 태블릿 세로에서 접힘·펼침 토글
- 게시판 메뉴에 읽지 않음 뱃지

[원칙 반영]
- 현재 페이지가 사이드 네비에서 명확히 강조 (원칙 2)
- 헤더 높이·여백은 모든 페이지에서 동일 (원칙 6)
- 사이드 네비 항목 터치 영역 충분히 (원칙 5)
- 피드백·알림 아이콘은 의미 색 토큰 사용 (원칙 6)

## 프롬프트가 12개인 이유

고객 4 + 어드민 7 + 공통 셸 1 = 12. 이 12개를 시안으로 잡으면 우리 서비스의 90% 화면이 커버된다. 게시판·설정·피드백 관리는 운영자가 가끔 보는 화면이라 어드민 공통 셸(S1)이 잡히면 그 안에서 자연스레 풀린다.

작업 순서는 A1(주문 관리 리스트) 후 A2(주문 상세 드로어)부터 시작한다(D3 결정). 그 두 화면이 끝나면 나머지가 어디로 갈지가 거의 자동으로 정해진다.
