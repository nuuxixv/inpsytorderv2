# 04. Claude 디자인 프롬프트

> 이 문서는 다른 자연어 문서들과 톤이 다르다. 절반은 자연어, 절반은 기능형이다. 각 프롬프트 앞에는 "이 화면이 누구의 어떤 순간인가"를 한두 문장 자연어로 두고, 그 아래에는 시안 도구(Claude Design / /design-html / /design-shotgun)에 그대로 붙여 넣을 수 있는 정밀 형식 본문을 둔다. 도입부는 작업자가 시안을 보기 전에 머리에 넣어두는 한 줄이고, 본문은 도구가 알아듣는 언어다. 둘을 섞은 이유는, 자연어만 두면 도구가 시안을 못 그리고 형식만 두면 작업자가 "왜 이렇게 그렸는지"를 잃기 때문이다.

## 왜 이 12개의 프롬프트를 모았는가

우리 서비스의 실제 화면은 13개다(고객 4개 + 어드민 9개). 그중 게시판·설정·피드백 관리·로그인은 어드민 공통 셸(S1)이 잡히면 같은 패턴 안에서 자연스레 풀린다. 그래서 시안 작업의 1차 대상은 12개 — 고객 4개(C1·C2·C3·C4), 어드민 7개(A1~A7), 공통 셸 1개(S1)다. 이 12개가 끝나면 서비스의 90% 화면이 디자인 시스템 위에 올라간다.

작업 순서도 임의가 아니다(D3 결정). **A1(어드민 주문 관리 리스트) → A2(어드민 주문 상세 드로어)부터 시작한다.** 가장 자주 쓰이고, 가장 많은 사고가 나는 화면이기 때문이다. 그 두 화면에서 토큰이 한 번 검증되면, 그다음 화면들은 같은 토큰을 빌려 쓰면서 빠르게 정리된다. 고객 화면 C1~C4는 어드민이 안정된 뒤 M4~M5에 들어간다.

## 어떻게 도구에 던지는가

각 프롬프트의 정밀 형식 본문 부분(대괄호 섹션부터 Do/Don't까지)을 그대로 복사해서 시안 도구에 넣는다. 자연어 도입부는 작업자가 읽는 용이고, 도구에 던질 필요는 없다. 변수 자리는 중괄호로 표시했다. 학회 이름, 상품 카테고리, 행사 슬러그 같은 건 작업 시점 데이터로 대체한다. 토큰 이름(brand-primary, space-lg 등)은 02 문서와 Appendix_Spec.md에 정의된 것 그대로 쓴다.

본문은 항상 다음 구조를 따른다 — [화면] [디바이스] [상황] [필수 요소] [원칙 반영] [Do] [Don't]. 도구가 읽기 좋은 순서이고, 작업자가 빠뜨린 게 있는지 한눈에 보이는 순서이기도 하다.

## 고객 화면 프롬프트 (C1 ~ C4)

### C1. 주문서 1단계 — 상품 선택

**이 화면이 누구의 어떤 순간인가.** 01에서 말한 첫 번째 주인공, 학회장 의자에 앉은 50대 의사가 부스 직원이 건넨 QR을 막 찍어서 들어온 순간이다. 이 사람에게 우리 서비스는 처음 보는 사이트이고, 본인 카드에서 돈이 빠진다. 30~80개 상품 중 보통 1~3개만 사면 끝나는데, 그 1~3개를 찾는 동안 가격이 헷갈리거나 화면이 부산스러우면 그분의 손이 멎는다. 이 시안에서 우리가 풀어야 할 것은 단 하나 — **본인 돈 쓰는 사람의 불안을 첫 화면에서 끝내는 일이다.**

```
[화면] 고객 주문서 1단계, 상품 선택
[디바이스] 모바일 우선 (iPhone 13 mini 375×812 기준), 태블릿 768까지 대응
[상황] 학회장 부스 앞에서 QR 진입. 30~80개 상품 중 1~3개 선택. 학회 슬러그가 URL에 포함됨.

[필수 요소]
- 상단 고정 헤더: 학회 이름(예 "한국심리학회 2026 추계") + 단계 표시("1/3 상품 선택")
- 검색바: 상단 헤더 아래, 폼 요소이므로 color-scheme: light 강제
- 카테고리 칩 가로 스크롤: "검사", "도서", "세트", "전체"
- 정렬 드롭다운: "인기순"(기본), "신상품순", "할인율순"
- 상품 그리드 (모바일 1열, 태블릿 2열):
  - 썸네일 (1:1)
  - 상품명 (title-card 18/600)
  - 가격 두 줄 위계:
    - 정가 (small 14/400, text-tertiary, 취소선)
    - 할인가 (title-section 20/700, text-primary)
  - 할인율 뱃지 (status-cancelled-soft 배경 + status-cancelled-bold 텍스트, 절제된 톤)
  - 담기 버튼 (md 사이즈, 우하단)
- 하단 고정 장바구니 바:
  - 담은 개수 + 합계 금액 (body-strong)
  - 무료배송 임계금액 미달 시: "OOO원 더 담으면 무료배송"
  - "주문서 작성" CTA (lg, brand-primary)

[원칙 반영]
- 원칙 1: 가격 두 줄 위계 (정가 작고 옅게, 할인가 크고 진하게)
- 원칙 1: 무료배송 임계금액 안내 (유도성 문구 금지 — 정보 제공만)
- 원칙 2: 본문 16px, 색은 의미 단위로만
- 원칙 5: 터치 영역 44×44px, 폼 요소 color-scheme: light
- 원칙 6: 카드 라운드 radius-md(10px), shadow-card 한 단계

[Do]
- 할인율 뱃지는 의료 톤 (붉은 배경 + 어두운 글자, 채도 한 단계 누름)
- 카테고리 칩의 활성 상태는 brand-primary-soft 배경
- 빈 상태(검색 결과 0): "검색 결과가 없습니다" + "전체 상품 보기" 버튼

[Don't]
- 정가와 할인가 같은 크기 금지
- 할인율 뱃지에 그라데이션·반짝임·애니메이션 금지
- "지금 사세요", "한정 수량" 같은 유도성 카피 금지
- 상품 카드 안에 다시 카드를 넣지 않음
```

### C2. 주문서 2단계 — 배송 정보 입력

**이 화면이 누구의 어떤 순간인가.** 같은 50대 의사가 상품을 골랐고, 이제 본인 정보를 입력해야 한다. 모바일 입력 폼은 누구에게나 어렵고, 안경 너머로 보는 그분에게는 더 그렇다. 게다가 검사 상품을 골랐다면 인싸이트 ID라는 낯선 필드까지 들어온다. 잘못 입력하면 학회 끝나고 운영자가 그분께 전화해야 한다. 이 시안에서 풀어야 할 것은 **라벨이 한 번도 사라지지 않게 만드는 일과, 인싸이트 ID 같은 낯선 필드의 이유를 한 줄로 알려주는 일이다.**

```
[화면] 고객 주문서 2단계, 배송 정보
[디바이스] 모바일 우선 (375×812)
[상황] 1단계 상품 선택 완료 후 진입. 본인 정보 입력.

[필수 요소]
- 상단: "2/3 배송 정보" + 뒤로가기 화살표
- 입력 필드 (각각 라벨이 input 위에 별도 행):
  - 이름 (필수)
  - 연락처 (필수, 휴대폰 패턴)
  - 인싸이트 ID (검사 상품 담긴 경우만 노출, "검사 결과를 받기 위해 필요합니다" 보조 텍스트)
  - 우편번호 (검색 버튼으로 다음 주소 모달 호출)
  - 도로명주소 (자동 채움, 읽기 전용)
  - 상세주소 (필수, 본인 입력)
  - 요청사항 (선택, textarea, 4행)
- 모든 input: height 48px, color-scheme: light 강제, border-focus 2px brand-primary
- 필수 표시: "(필수)" 텍스트 (빨간 별표 금지)
- 에러 메시지: 필드 바로 아래, "왜 안 됐는지 + 무엇을 하면 되는지" 한 줄
- 하단 고정 CTA: "다음 단계로" (lg, brand-primary)

[원칙 반영]
- 원칙 4: 라벨은 input 위에 항상 표시 (placeholder 안에 라벨 금지)
- 원칙 4: 에러 메시지는 진단 + 처방 둘 다
- 원칙 5: 본문 16px, 터치 44×44px
- 원칙 2: 인싸이트 ID는 검사 상품 담긴 경우만 (필요 없을 때 안 보이게)

[Do]
- 라벨 클릭 시 input에 포커스
- 우편번호 검색 모달은 BottomSheet 형태 (모바일)
- input 비활성 시 surface-3 배경, 비활성 이유 보조 텍스트

[Don't]
- placeholder 안에 라벨 두는 패턴 금지
- 한 행에 input 2개 나란히 배치 금지 (모바일 가독성)
- 자동완성 기능 추가 금지 (개인정보 이슈)
```

### C3. 주문서 3단계 — 최종 확인

**이 화면이 누구의 어떤 순간인가.** 결제 직전이다. 본인 돈 쓰는 사람이 마지막으로 멈춰서 "정말 이 금액이 맞나"를 묻는 순간. 이 화면에서 의심이 풀리지 못하면 결제 자체가 일어나지 않는다. 그리고 우리는 출고 알림이 없기 때문에, **배송 예정일이 이 화면에서 명확히 보여야 한다.** 부스 직원에게 카드를 건네는 흐름이라는 것도 한 줄 안내가 필요하다.

```
[화면] 고객 주문서 3단계, 최종 확인
[디바이스] 모바일 우선

[필수 요소]
- 상단: "3/3 주문 확인" + 뒤로가기
- 주문 상품 요약 카드: 상품명·수량·금액, 접고 펴기 가능 (기본 펼침)
- 배송지 요약 카드: 이름·연락처·주소, 우상단 "수정" 텍스트 버튼
- 금액 정리 섹션 (가로 구분선으로 분리):
  - 상품 금액
  - 할인 (-)
  - 배송비 (무료배송 임계금액 안내 한 줄)
  - 합계 (title-section 20/700 또는 그 이상, 가장 큰 글씨)
- 배송 예정일 카드: "학회 종료 후 D+7일 이내 발송 예정" 강조 (title-card)
- 약관 동의 체크박스 (필수): "주문 내용을 확인했으며 결제에 동의합니다"
- 하단 고정 CTA: "결제하러 가기" (lg, brand-primary)
- CTA 위 한 줄 안내: "결제는 부스 직원에게 카드를 건네 진행됩니다"

[원칙 반영]
- 원칙 1: 합계 금액이 화면에서 가장 큰 글씨
- 원칙 1: 배송 예정일이 명확히 보임
- 원칙 2: 색은 의미 단위 — brand-primary는 CTA 한 자리만
- 원칙 3: "결제하러 가기"는 brand 색, 그 외 버튼은 약하게

[Do]
- 상품 요약 카드는 접혀 있어도 합계는 항상 보임
- "수정" 텍스트 버튼은 1단계·2단계로 돌아가는 경로
- 무료배송 임계금액 안내는 정보 제공 톤 ("OOO원 더 담으면 무료배송")

[Don't]
- "마지막 기회", "지금 결제" 같은 유도성 카피 금지
- 합계 금액을 작게 두는 디자인 금지
- 배송 예정일을 숨김 처리 금지
```

### C4. 주문 조회 페이지

**이 화면이 누구의 어떤 순간인가.** 결제가 끝났고, 알림톡이 왔고, 그 알림톡 안의 링크를 누른 사람이 도착하는 곳이다. 우리는 출고 알림을 보내지 않기 때문에 이 페이지가 **사실상 유일한 상태 확인 창구**다. "내가 산 게 뭐였지, 언제 오지"가 0.5초 안에 답이어야 한다. 그래서 결제완료 알림톡 한 통의 무게가 무겁다고 01에서 말한 것이다 — 이 페이지가 그 무게를 받친다.

```
[화면] 고객 주문 조회 페이지
[디바이스] 모바일·태블릿·PC 모두 대응
[상황] 알림톡 버튼 또는 QR로 진입. UUID access_token 기반 단건 조회.

[필수 요소]
- 상단: 학회 이름 + 주문 번호 (작게)
- 주문 상태 큰 라벨: 결제완료/배송준비/배송완료 등 (title-section, status- 토큰 색)
- 배송 예정일: 큰 글씨로 강조 ("학회 종료 후 D+7일 이내 발송 예정")
- 주문 상품 목록: 썸네일·상품명·수량·금액
- 배송지 요약 (읽기 전용, "정정이 필요하면 부스 또는 채널톡으로 문의" 안내)
- 결제 정보: 상품 금액 / 할인 / 배송비 / 합계
- 문의 안내: 인싸이트 카카오 채널 또는 부스 연락처

[원칙 반영]
- 원칙 2: 주문 상태와 배송 예정일이 화면 상단에 가장 크게
- 원칙 6: 상태 색은 status- 토큰만, 어드민과 동일 의미·색
- 원칙 1: 본인 돈 쓴 사람의 "언제 오지" 답이 명확

[Do]
- 상태 라벨 옆에 알림 아이콘 + "결제완료 알림톡이 발송되었습니다" 보조 텍스트
- "배송 예정일은 학회 종료 후 일괄 발송 기준입니다" 한 줄 설명

[Don't]
- "출고 알림이 없습니다" 같은 부정 안내 금지 (대신 "학회 종료 후 일괄 발송" 긍정 표현)

    - 상태 칩 (status-active·upcoming·ended)
    - 액션 버튼 4종:
      - "주문 URL 새 창" (외부 링크 아이콘)
      - "URL 복사" (복사 아이콘 → 토스트)
      - "QR SVG 다운로드" (다운로드 아이콘)
      - "편집" (펜 아이콘 → 편집 모달)
- 신규 행사 추가 모달:
  - 이름 / 슬러그(자동 생성, 편집 가능) / 시작일·종료일 / 할인율(%) / 배송 예정일(D+N)
  - 주최 학회 셀렉트 / 시즌 셀렉트
  - 저장 / 취소
- 학회 탭: 학회 CRUD (이름·연락처·메모)
- 시즌 탭: 시즌 CRUD (이름·기간)
- 카테고리 매핑 탭: 행사별 노출 카테고리 설정

[원칙 반영]
- 원칙 2: "진행중" 상태 칩이 가장 강조
- 원칙 4: URL 복사·QR 다운로드는 한 탭 + 토스트 피드백
- 원칙 3: 행사 삭제는 별도 위험 액션 (편집 모달 안에 빨간 버튼)
- 원칙 6: 상태 색은 status- 토큰 통일

[Do]
- 슬러그는 한글 → 영문 자동 변환 후 수정 가능
- 진행중 행사를 리스트 상단에 자동 정렬
- QR SVG는 즉시 다운로드 (모달 없이)

[Don't]
- 행사 삭제에 받아쓰기까지 요구하지 않음 (편집 모달 안 확인 모달 1회면 충분)
- 카테고리 매핑을 자동화하지 않음 (마스터 의도로만)
```

### A6. 어드민 상품 관리

**이 화면이 누구의 어떤 순간인가.** 운영자가 가장 무서워하는 화면이다. 상품 수백 개, 엑셀 업로드, 다중 선택, 일괄 편집, 전체삭제까지. 한 번 잘못 누르면 학회 직전 상품 카탈로그가 비어버릴 수 있다. 그래서 이 화면의 핵심은 **위험 액션의 차단 — 자주 하는 일은 빠르게, 되돌릴 수 없는 일은 받아쓰기까지 거치게**이다.

```
[화면] 어드민 상품 관리 페이지
[디바이스] 데스크탑·태블릿 가로

[필수 요소]
- 상단 헤더:
  - 페이지 제목 "상품 관리"
  - 검색바 (상품명·SKU)
  - 카테고리 필터 (도서 / 검사 / 세트 / 전체)
  - "신규 상품 추가" 버튼 (primary)
  - "엑셀 업로드" 버튼 (secondary)
- 필터 칩: 활성 / 비활성, 재고 있음 / 없음
- 테이블:
  - 체크박스 / 썸네일 / 상품명 / SKU / 카테고리 / 정가 / 할인가 / 재고 / 활성 토글 / 액션(편집·삭제)
- 선택 액션 바 (1개 이상 선택 시):
  - "N건 선택" + 일괄 편집 / 선택 삭제 (확인 모달 1회)
- 엑셀 업로드 모달:
  - 파일 드래그·선택 영역
  - 업로드 시작 후: 진행률 바 / 청크별 상태 / 행 단위 오류 보고 (어떤 행이 왜 안 됐는지)
- 위험 영역 (페이지 하단 별도 카드):
  - "전체 상품 삭제" 버튼 (status-cancelled 톤)
  - 클릭 시 모달: "전체삭제"라고 정확히 타이핑해야 진행
  - 입력 일치 시에만 삭제 버튼 활성

[원칙 반영]
- 원칙 3: 자주 하는 일(편집·일괄 활성/비활성)은 두 탭
- 원칙 3: 전체삭제는 받아쓰기 (의도적으로 느리게)
- 원칙 4: 엑셀 업로드 오류는 행 단위 진단 + 처방
- 원칙 4: 빈 상태에 엑셀 업로드 안내

[Do]
- 활성 토글은 즉시 반영 + Toast (Undo 5초 가능)
- 엑셀 업로드 청크 진행은 화면에서 확인 가능
- 일괄 편집은 가격·재고·활성 한정 (상품명·SKU 일괄 편집은 위험하므로 금지)

[Don't]
- 전체삭제에 받아쓰기 없는 단순 확인 모달 금지
- 일괄 삭제에 Undo 제공 금지 (이미 받아쓰기로 의도 확인됨)
- 엑셀 업로드 오류를 토스트로만 보여주지 않음 (행 단위 패널 필수)
```

### A7. 어드민 사용자 관리 + 역할 템플릿

**이 화면이 누구의 어떤 순간인가.** 마스터가 새 운영자 계정을 만들거나, 역할 권한 매트릭스를 손보는 순간이다. 자주 만지지 않지만, 만질 때 분명히 보여야 한다. master 권한이 잘못 부여되면 운영 전체가 흔들린다. 그래서 매트릭스 UI는 충분히 크고, master 권한 부여·해제는 별도 위험 액션이다.

```
[화면] 어드민 사용자 관리 페이지
[디바이스] 데스크탑·태블릿 가로

[필수 요소]
- 상단 탭: 사용자 목록 / 역할 템플릿
- 사용자 목록 탭:
  - 검색바 (이름·이메일)
  - 신규 사용자 추가 버튼
  - 테이블: 이름 / 이메일 / 역할 칩 / 마지막 로그인 / master 여부 아이콘 / 액션
  - 액션: 역할 변경 / 비밀번호 초기화 / 비활성화 / 삭제
  - master 권한 부여·해제는 별도 위험 액션 (색·위치 분리, 확인 모달)
- 역할 템플릿 탭:
  - 매트릭스 UI:
    - 행: 역할 (master, manager, operator, viewer 등)
    - 열: 권한 (orders.read / orders.write / products.write / users.admin 등)
    - 셀: 체크박스 (44×44 터치 영역)
  - "신규 역할 추가" 버튼
  - 역할 편집·삭제 (사용 중인 역할은 삭제 차단)
- 권한 변경 저장 시 확인 모달 1회

[원칙 반영]
- 원칙 5: 매트릭스 셀 44×44px 이상
- 원칙 3: master 권한 부여·해제는 별도 위험 액션
- 원칙 4: 사용 중인 역할 삭제 차단 + 이유 안내
- 원칙 2: master 사용자는 시각적으로 구분 (아이콘 또는 색)

[Do]
- 매트릭스 셀 호버 시 행·열 동시 강조
- 권한 변경 후 저장 전에는 "저장하지 않은 변경" 표시
- 역할 삭제 시 해당 역할 부여된 사용자 수 표시

[Don't]
- 매트릭스 셀을 작게 만들지 않음 (잘못 누르면 권한 사고)
- master 권한 부여를 일반 역할 변경과 같은 자리에 두지 않음
- 비밀번호 초기화에 토스트만 띄우지 않음 (모달로 새 비밀번호 표시)
```

## 공통 셸 프롬프트 (S1)

### S1. 어드민 공통 셸 — 헤더 + 사이드 네비

**이 화면이 누구의 어떤 순간인가.** 모든 어드민 페이지가 이 셸 안에 들어간다. 셸이 흔들리면 페이지 사이 일관성이 무너진다. 게시판 읽지 않음 뱃지, 알림 아이콘, 피드백 제출, 사용자 메뉴가 헤더에 산다. 셸은 사용자가 의식하지 않지만, **잘못 만들면 모든 화면이 잘못 보인다.**

```
[컴포넌트] 어드민 공통 셸 — 헤더 + 사이드 네비
[디바이스] 태블릿 세로 1차 + 데스크탑

[필수 요소]
- 상단 헤더 (sticky, 높이 64px):
  - 좌측: 로고 + 현재 페이지 제목
  - 중앙: 전역 검색 (옵션, MVP에선 생략 가능)
  - 우측 아이콘 행:
    - 피드백 제출 버튼 (말풍선 아이콘)
    - 알림 아이콘 (벨, 새 알림 시 빨간 점)
    - 사용자 메뉴 (이름 + 드롭다운 화살표, 로그아웃 포함)
- 좌측 사이드 네비 (sticky, 너비 240px 펼침 / 64px 접힘):
  - 메뉴:
    - 대시보드
    - 주문 관리
    - 출고 현황
    - 학회 관리
    - 상품 관리
    - 사용자 관리 (master만)
    - 게시판 (읽지 않음 뱃지)
    - 설정
    - 피드백 관리 (master만)
  - 각 항목: 아이콘 + 라벨 + 활성 강조 (좌측 brand-primary 라인 + brand-primary-soft 배경)
- 태블릿 세로(820px): 사이드 네비 기본 접힘, 햄버거 버튼으로 펼침
- 데스크탑(≥1280): 사이드 네비 기본 펼침

[원칙 반영]
- 원칙 2: 현재 페이지가 사이드 네비에서 명확히 강조
- 원칙 6: 헤더 높이·여백·로고 위치가 모든 페이지에서 동일
- 원칙 5: 사이드 네비 항목 터치 영역 충분 (높이 48px+)
- 원칙 6: 피드백·알림 아이콘은 status- 토큰 사용

[Do]
- 사이드 네비 접힘 상태에서도 아이콘 + 툴팁(호버) 동작
- 게시판 읽지 않음 뱃지는 숫자 표시 (10건 이상은 "9+")
- 알림 아이콘 클릭 시 드롭다운 (최근 5건 + "모두 보기")
- 활성 메뉴는 URL 기반 자동 판별

[Don't]
- 헤더에 광고·배너 영역 두지 않음
- 사이드 네비에 메뉴 10개 이상 두지 않음 (현재 9개가 최대)
- 사용자 메뉴에 설정 중복 두지 않음 (사이드 네비의 설정으로 통일)
```

## 프롬프트가 12개인 이유

고객 4 + 어드민 7 + 공통 셸 1 = 12. 이 12개가 시안으로 잡히면 우리 서비스 화면의 90%가 디자인 시스템 위에 올라간다. 나머지 4개 페이지 — 게시판, 설정, 피드백 관리, 로그인 — 는 S1 셸이 잡히면 그 안에서 같은 패턴으로 자연스레 풀린다. 게시판은 카드 리스트 + 상세 모달 패턴, 설정은 폼 패턴, 피드백 관리는 A1 리스트 패턴의 변형, 로그인은 별도 셸이지만 input + button + brand 강조의 단순한 화면이다. 이 4개는 별도 프롬프트 없이 M3~M5 사이에서 시안 정리 단계에 묶어 처리한다.

작업 순서는 D3 결정대로 **A1 → A2가 먼저**다. 그 두 화면이 끝나면 토큰이 한 번 검증되고, 나머지 화면들은 같은 토큰을 빌려 쓰면서 빠르게 정리된다. M2(2주)에 A1·A2를 잡고, M3(3주)에 어드민 나머지 + 셸, M4(3주)에 어드민 코드 적용 + 고객 시안, M5(2주)에 고객 코드 적용 — 이 흐름이 07 로드맵의 핵심이다.
