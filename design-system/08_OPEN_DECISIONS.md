# 08. 결정 사항 — D1부터 D14까지

> 디자인 시스템 재설계 시작부터 지금까지 내려진 결정들을 한 페이지에 모은다. 확정된 것과 아직 비어 있는 것을 구분해서 본다. 이 페이지를 보면 "지금 어디까지 정해졌는가"가 한눈에 풀린다.

## 확정된 결정 14건

### D1. 토스식 브랜치 처리

`feature/toss-style-ux-overhaul` 브랜치는 디자인 시안으로 그대로 가져오지 않는다. 그 안의 비-디자인 개선(버그 수정, 작은 기능 보강)만 골라 main으로 체리픽한다. 디자인은 새 브랜치에서 처음부터.

**왜.** 토스 답습이 절반 실패라는 평가. 우리 사용자(50대 의사·부스 운영자)에게 안 맞는 부분이 너무 많다.

### D2. 검수 체계

CPO 1차 검수 후, 마일스톤 종료 시점에 건우님이 공동 승인. M1 원칙 확정에는 건우님 필수 참여.

**왜.** CPO 혼자 결정하면 톤이 흔들릴 수 있고, 매번 건우님이 보면 속도가 안 난다. 절충.

### D3. 시안 시작점

어드민 주문 관리 페이지(A1 리스트, A2 상세 드로어)부터 시안 착수.

**왜.** 운영자가 가장 자주 보는 화면이라 가장 빨리 새 시스템으로 옮겨야 하고, 가장 복잡해서 토큰의 한계가 가장 빨리 드러난다.

### D4. 검수 단계 수

3단 게이트. 원칙 정합성(시안 직후), 시스템 일관성(시안 모음), 실기기 QA(코드 적용 후).

**왜.** 한 번에 다 보면 놓친다. 단계마다 보는 눈이 달라야 잡힌다.

### D5. 마일스톤 수

5개(M1~M5). 학회 D-4개월 역산.

**왜.** 마일스톤이 더 많으면 잘게 쪼개져 흐름이 끊기고, 더 적으면 검수가 무거워진다.

### D6. 자연어 톤

모든 디자인 문서는 표·수치 중심이 아닌 **자연어 중심**으로 쓴다. 픽셀 단위 수치는 Appendix_Spec.md에 분리.

**왜.** 디자인 결정의 맥락이 문장으로 전달되어야 다음 사람이 같은 톤을 이어간다. 표만 있으면 톤이 사라진다.

### D7. Figma 활용 시점

Figma는 연결 대기. 연결 후 **현재 UI 리버스 임포트** 중심으로 시작. Variables 선제 구축 안 함.

**왜.** Variables 먼저 만들면 시안 한 번 만들고 다시 깔아엎게 된다. 시안이 어느 정도 자리 잡은 다음 Figma가 그걸 따라온다.

### D8. Notion 미러링 안 함

Notion에 디자인 문서를 따로 옮기지 않는다. design-system/*.md를 클로드 코드 안에서 직접 읽는다.

**왜.** Notion 미러링은 동기화 비용이 들고, 결국 어느 쪽이 정본인지 헷갈린다. MD 파일이 정본.

### D9. Figma 리버스 임포트 우선

(D7 보강) 시안 도구에서 만든 결과를 Figma에 임포트하는 도구를 통해 Figma 파일을 채워간다. Figma에서 시작해 코드로 가는 게 아니라 반대 방향.

**왜.** 우리 팀이 Figma 사용에 익숙하지 않고, 클로드 시안 출력이 더 빠르다. Figma는 결과물 보관소 역할에 가깝다.

### D10. 다크·고대비·큰글자 — 가시성 세이프가드 수준

별도 다크 테마·고대비 토글·큰글자 모드 모두 폐기. `prefers-color-scheme: dark`, `prefers-contrast: more`, 시스템 줌 150% 대응만 한다.

**왜.** 우리 사용자는 이런 토글을 찾아 누르지 않는다. 시스템 설정을 켠 사용자에게 자동으로 잘 보이게 해주는 게 ROI가 훨씬 크다. 자세한 건 09 문서.

### D11. 브랜드 메인 색 — 인싸이트 실제 브랜드

`brand-700` = `#2B398F` (인싸이트 진한 남색). 명도 단계로 `brand-900`(`#1a237e`) / `brand-700`(메인) / `brand-500`(`#3d4db0`) / `brand-300` / `brand-100` 다섯 단계.

**왜.** 인싸이트·학지사가 오래 써온 실제 브랜드 색이 `theme.js`에 이미 박혀 있었다(`PRIMARY_MAIN = #2B398F`). CPO가 한때 후보로 올렸던 청록 계열(`#0F766E`, `#0E7490` 등)은 모두 추측이었고 폐기. 추측 후보 대신 사용자가 인싸이트 부스에서 매년 본 적 있는 색을 그대로 잇는 게 신뢰감 면에서 ROI가 압도적이다.

### D12. 페이지 제목 크기 — 24px

`title-page` = 24 / 1.2 / 700.

**왜.** 기존 `theme.js`는 28(`1.75rem`). 어드민 태블릿 세로 820 화면에서 24가 이미 충분히 큰 위계를 만들고, 모바일 고객 화면에서 28은 한 줄을 잡아먹어 부산스러웠다. "너무 클 필요는 없어 보인다"는 건우님 판단을 반영. 본문 16과의 비율(1.5배)도 자연스럽다.

### D13. 버튼 라운드 — 8px

`radius-button` = 8.

**왜.** 카드(`radius-md` = 10)보다 한 단계 작아야 버튼이 카드 안에 단정하게 앉는다. 버튼이 카드보다 둥글면 "뚫고 떠 있는" 느낌. 기존 `theme.js`의 12(일반)·14(large)는 토스식 캡슐화 흔적이라 8로 통일.

### D14. 공용 `<Text>` wrapper — 도입 안 함

별도 `components/ui/Text.jsx` wrapper를 신설하지 않는다. 모든 텍스트는 MUI `<Typography variant="...">`로 호출한다. 같은 표시 패턴이 반복되는 자리(라벨+값의 `InfoRow`, 칩 안의 라벨 등)는 wrapper가 아니라 **합성 컴포넌트**로 추출한다.

**왜.** 우리가 풀어야 할 문제는 "wrapper가 없어서"가 아니라 "약속이 없어서"였다. M2 시안 9개 합 인라인 `fontSize` 332건 / 토큰 호출 0건의 원인은 변환 도구가 부족해서가 아니라 작업자에게 부르는 약속이 없었기 때문이다. 02 §타이포 §사용 규칙으로 약속을 박는 게 1차 처방. wrapper를 한 겹 더 끼우면 (1) 컴포넌트 의존성이 한 겹 늘고, (2) 시안 작업자가 MUI Typography 외에 추가로 배워야 할 단어가 생기고, (3) variant 확장 vs 새 prop 두 갈래로 일관성이 더 나빠진다. 약속이 충분히 정착한 다음에도 wrapper가 필요한 패턴이 발견되면, 그건 Text wrapper가 아니라 의미 단위 합성 컴포넌트(`InfoRow`, `KeyValue`, `StatusLabel` 등)로 만든다. 그쪽이 우리 결에 맞다.


## 아직 비어 있는 것

- **본문 폰트 스택의 최종 순서.** Pretendard Variable 우선 / 시스템 폰트 fallback 순서는 현재 `theme.js` 기준대로 유지 중이지만, 웹폰트 로딩 비용과 학회장 네트워크 환경을 고려한 최종 확정은 M2 시점.
- **어드민 사이드 네비의 메뉴 순서.** 현재 9개. 정보 설계(IA) 재검토 항목이라 시점 미정. 일단 현재 순서 유지.
- **부분 취소 UX.** 특정 상품만 취소 시 화면 흐름 미정. 디자인 시스템 작업과 별개로 진행될 수도, 마지막 사이클에 합쳐질 수도 있음.
- **다크모드 brand-700 대치 정확값.** §11.2에 `#5E6CC8` 근방으로 명시했지만 단일 헥사 확정은 가시성 세이프가드 CSS 패치 시점.
- **M2 시안 백로그.** 1순위 출고 현황(A3)은 본 마일스톤 내 진행. 2순위 상품 관리(A6, 전체삭제 받아쓰기 + 엑셀 진행률 패턴), 3순위 학회 관리(A5, 행사 카드 + 액션 4종)는 M2 후반 또는 M3 초반에 채택안 톤 검증 결과를 반영해 진행. 사용자 관리·피드백·게시판은 S1 공통 셸 정착 이후 M3 후반 일괄 처리.
