# specs — 화면 사양 시트 모음

이 폴더는 디자인 시안과 실서비스 구현이 같은 이야기를 하게 만드는 자리다. 시안이 실 페이지의 정보 구조를 임의로 단순화·통합·생략하지 않도록, 각 화면의 표시 정보·액션·입력 폼·DB 스키마를 라벨 단위로 정리한 단일 진실 소스를 둔다.

## 왜 이 폴더가 필요한가

2026-05-13에 우리는 출고 시안에서 세 가지를 한꺼번에 발견했다. 주소가 도로명·상세·우편번호 세 필드로 입력받는 구조인데 시안에서는 한 줄로 통합돼 있었고, 상품 행에 있어야 할 카테고리 칩(도서/검사)이 빠져 있었고, 비슷한 의미의 정보들이 서로 다른 서식으로 흩어져 있었다. 세 건 모두 시안 단계에서 잡혔어야 할 일이다. 잡지 못한 이유는 단순했다 — 시안 작업자가 실 페이지의 정보 구조를 옆에 두고 작업하지 않았기 때문이다.

`specs/`는 그 옆에 두는 자리다. 시안을 만들기 전에 사양 시트를 펴고, 만든 다음에 사양 시트의 모든 항목이 시안에 존재하는지 체크한다.

## 사용 절차

`design-system/06_REVIEW_PROCESS.md` 의 **게이트 1.5 — 기획 충실도** 와 함께 본다. 그리고 프로젝트 루트 `CLAUDE.md` 의 **기획 충실도 룰** A·B·C·D·E와 1:1 매칭된다.

1. **시안 작업 시작 전.** 해당 화면의 사양 시트를 펴고 표시 정보·액션·입력 폼 구조를 머리에 넣는다. 시트가 없으면 먼저 만든다(아래 참고).
2. **시안 직후.** 사양 시트의 모든 체크리스트 항목이 시안에 존재하는지 한 줄씩 확인한다. 누락·임의 통합·임의 변경이 한 건이라도 발견되면 게이트 1.5 불통과.
3. **구현 위임 시.** 메인 Claude가 위임 지시서에 사양 시트의 항목을 인용해서 박는다. frontend 보고는 사양 시트 항목 N개 중 시안 반영 M개 형식의 1:1 매핑 표를 의무로 동반한다.
4. **사양 변경이 정말 필요할 때.** 시안이 아니라 사양 시트를 먼저 고친다. 사양 시트 변경은 건우님 승인 사안. 이력란에 일자와 사유를 남긴다.

## 새 화면 사양 시트 추가 방법

1. `_TEMPLATE.md` 를 복사해서 `{화면코드}_{화면명}.md` 로 만든다(예: `A1_OrderManagementPage.md`).
2. 실 페이지(`*Page.jsx`)와 관련 API, 관련 입력 폼, 그리고 `supabase/migrations/` 의 해당 테이블 SQL을 정독한다.
3. 표시 정보·액션·입력 폼 구조를 **실 코드에 박혀 있는 사실만** 옮긴다. 추측은 금지. 확인 불가 항목은 "확인 필요"로 표기.
4. 사양 시트 자체에 대한 검수는 메인 Claude의 sampling check로 이뤄진다(CLAUDE.md C 절차).

다른 화면의 시트가 필요하면 `_TEMPLATE.md` 를 복사해 같은 톤·포맷으로 추가한다.

## 톤 약속

사양 시트의 본문은 **체크리스트 형식**으로 둔다. 다른 디자인 시스템 문서(01·06 등)는 자연어 톤을 따르지만, 사양 시트만은 빠진 항목이 한눈에 보이도록 체크박스 한 줄 한 줄로 적는다. 사고를 막는 도구이지 읽기 좋은 산문이 아니다.
